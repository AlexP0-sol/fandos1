## 1. Механизм

На перпетуальных фьючерсах каждые 1–8 часов происходит **funding** — платёж между лонгами
и шортами. Ставка на разных биржах по одному активу различается. Бот:

1. **Сканирует** все пары бирж по каждому активу и считает ожидаемый чистый доход:

   ```
   ExpectedNetPnL = funding-доход (short-нога получает больше, чем платит long-нога)
                  + базис (разница цен входа)
                  − комиссии входа и выхода
                  − проскальзывание
                  − резервы (неопределённость funding, расхождение базиса,
                    контрагентский риск, safety)
   ```

2. Если `ExpectedNetPnL ≥ порога` и все фильтры пройдены — **одновременно** открывает
   long на бирже с низкой ставкой и short на бирже с высокой. Позиция **дельта-нейтральна**:
   рост/падение цены актива на PnL не влияет.
3. **Сопровождает**: следит за дельтой, funding-спредом, маржой, ADL.
4. **Закрывает** обе ноги синхронно — по вашей команде или когда funding-спред развернулся.

Доход = собранные funding-платежи + базис − издержки. Главный риск — не цена, а
**расхождение ног** (одна закрылась, другая нет) — против этого направлена вся механика
исполнения (см. раздел 5).

## 2. Состав

| Процесс | Что делает | Порты |
|---|---|---|
| **worker** | рыночные данные → сканер → движок исполнения → риск → outbox | `:9090` метрики/health |
| **server** | Mini App + HTTP API + Telegram-бот | `:8080` |
| **PostgreSQL** | единственный источник истины: позиции, ордера, ключи, аудит | 5432 |

Процессы общаются **только через БД** (transactional outbox) — можно рестартовать любой
независимо, состояние не теряется.

## 3. Mini App

Открывается из Telegram-бота (кнопка меню) или в браузере по адресу server (в браузере —
демо-режим с моковыми данными). Вход: Telegram проверяет подпись `initData`, сервер сверяет
ваш id с allowlist (`TELEGRAM_ADMIN_IDS`) и выдаёт сессию на 24 часа. **Первый вошедший
allowlisted-пользователь автоматически становится владельцем** системы.

### 3.1 Dashboard (главный экран)

- **Бейдж состояния**: `RUNNING` (зелёный) — торговля разрешена; `SAFE_HALT` (красный) —
  новые входы запрещены (причина — в уведомлениях/логах).
- **Режим**: `dry_run` / `live`.
- **Карточки бирж**: equity, свободная маржа, состояние подключения (REST/WS).
- **Открытые позиции**: счётчик + список.

### 3.2 Candidates (кандидаты сканера)

Таблица пар, отсортированная по ожидаемой доходности:
актив, long-биржа, short-биржа, **ExpectedNetPnL** (раскрывается в полную разбивку — все
резервы и комиссии), composite score, уверенность (confidence), секунды до ближайшего funding.

> Кандидаты — транзиентные данные worker-а. Сервер отдаёт их из своего снапшота; в браузерном
> демо-режиме показываются примеры. «Живые» кандидаты всегда видны в логах worker
> (`msg="eligible candidate"`).

### 3.3 Positions (позиции)

Каждая позиция: id, актив, ноги (биржа/объём long и short), состояние state machine
(бейдж), дельта, funding-PnL, комиссии. Кнопка **Close** — запрос на координированное
закрытие (уходит через outbox в worker; исполняется в течение нескольких секунд).

### 3.4 Settings (настройки)

Все HOT-настройки (раздел 4). Сохранение — с **optimistic-версией**: если кто-то изменил
настройки параллельно, получите конфликт 409 — перечитайте и повторите. Изменения
применяются **без рестарта**.

### 3.5 API-ключи

Список подключённых ключей (биржа, тип, отпечаток `AbCd...WxYz`, дата, статус
active/revoked) + форма добавления: биржа (все 7), тип (`trade` / `withdrawal`),
API key, API secret, passphrase (нужна только OKX/Bitget/KuCoin). Кнопка **Revoke**
отзывает ключ (бот перестаёт его использовать; сам ключ на бирже не удаляется — удалите
его в кабинете биржи).

Повторный ввод ключа той же биржи/типа = **ротация** (старый blob замещается, отзыв снимается).

### 3.6 KILL SWITCH

Красная кнопка. Подтверждение + 2FA-код (если включён). Действие раздела 7.2.

## 4. Настройки

Все настройки — HOT: хранятся в БД, правятся через Mini App, применяются на лету.
Значения по умолчанию — консервативные.

### 4.1 Стратегия поиска

| Настройка | По умолчанию | Что значит / рекомендации |
|---|---|---|
| `fundingSearchMode` | `SAME_INTERVAL` | Какие пары рассматривать: `SAME_INTERVAL` — только биржи с одинаковым интервалом funding (проще и предсказуемее). Расширенные режимы допускают разные интервалы — выше потенциал, выше неопределённость. |
| `requireAlignedFundingTimes` | `true` | Требовать, чтобы моменты funding на обеих биржах совпадали (в пределах допуска). `true` — funding-потоки синхронны, оценка точнее. |
| `minExpectedNetPnlUsdt` | `1` | Минимальный ожидаемый чистый доход (USDT) для входа — **после всех** комиссий и резервов. Поднимите, чтобы входить реже, но качественнее. |
| `minConfidenceLevel` | `2` (Medium) | Минимальная уверенность в predicted funding (0=None,1=Low,2=Medium,3=High). Уверенность растёт по мере приближения funding-момента. |
| `minSecondsBeforeFundingToEnter` | `30` | Не входить, если до funding меньше N секунд (не успеть набрать обе ноги). |
| `requireBacktestPass` | `true` | Требовать пройденный backtest-гейт перед live-входами. |

### 4.2 Риск и капитал

| Настройка | По умолчанию | Что значит / рекомендации |
|---|---|---|
| `leverage` | `2` | Плечо обеих ног. Держите низким (1–3): плечо здесь не про доходность, а про эффективность капитала; высокое плечо приближает ликвидацию ноги. |
| `marginMode` | `isolated` | Маржа: `isolated` — проблема одной позиции не трогает остальной счёт (рекомендуется), `cross` — общий пул. |
| `positionMode` | `one_way` | Режим позиций на бирже. Бот работает в one-way. |
| `maxDailyLossUsdt` | `0` = не задан | **Задайте обязательно.** Дневной лимит реализованного убытка; при превышении — риск-стоп. |
| `maxPositionLossUsdt` | `0` = не задан | Лимит убытка одной позиции → принудительное закрытие. |
| `riskSnapAfterMaxDailyLoss` | `true` | При пробое дневного лимита — SAFE_HALT (полная остановка входов), не просто «пропуск сделки». |
| `jointSlippageCapBps` | `20` | Совокупный допуск проскальзывания обеих ног, б.п. (20 = 0,2%). |
| `maxExposurePerExchangeUsdt` | — | Максимальный notional на одной бирже (контрагентский риск). Задайте по каждой бирже. |
| `deltaToleranceBase` / `deltaToleranceUsd` | — | Допустимый дисбаланс ног в базовой монете / USDT. Больший дисбаланс → repair, неустранимый → DEGRADED. |

### 4.3 Исполнение

| Настройка | По умолчанию | Что значит |
|---|---|---|
| `orderMode` | `marketable_limit_ioc` | Тип входных ордеров: marketable-limit IOC (лимит по цене «через спред» с защитой в тиках) — контроль worst-price. `market` — быстрее, но без ценовой защиты. |
| `ackTimeoutBehavior` | `QUERY_THEN_DECIDE` | Единственно верное поведение при таймауте подтверждения: **запросить состояние ордера**, а не слать повторно вслепую (защита от дублей). Не менять. |
| `orderAckTimeoutMs` | `3000` | Сколько ждать подтверждения ордера. |

### 4.4 Условия выхода

| Настройка | По умолчанию | Что значит |
|---|---|---|
| `exitIfAdlDetected` | `true` | Закрываться при попадании ноги в зону авто-делевериджа (ADL) биржи. |
| `exitIfFundingSignChanges` | `true` | Закрываться, когда funding-спред развернулся (позиция начала платить вместо получать). Основной автоматический выход. |

### 4.5 Ребалансировка (перевод средств между биржами)

| Настройка | По умолчанию | Что значит |
|---|---|---|
| `rebalanceEnabled` | `false` | Автоперевод USDT для выравнивания 50/50. **Держите выключенным**, пока не сверите withdrawal-эндпоинты вашей пары бирж и не одобрите адреса. |
| `withdrawalFeeCapUsdt` | — | Максимально допустимая комиссия одного вывода; дороже — план отменяется. |
| `withdrawalFailureThreshold` | `3` | Сбоев подряд до срабатывания withdrawal circuit breaker (дальнейшие выводы блокируются до ручного сброса). |
| `depositGracePeriodMs` | `1800000` (30 мин) | Сколько ждать зачисления, прежде чем считать перевод проблемным. |

Ребаланс двухэтапный: сначала **тестовый микро-перевод**, и только после его подтверждения —
основной. Адреса — только из заранее одобренного белого списка; в логи попадают только
отпечатки адресов.

## 5. Жизненный цикл

```
DISCOVERED → QUALIFIED → PREPARING → OPENING → [PARTIALLY_HEDGED] → HEDGED → MONITORING
                                                                                │
                                              EXIT_REQUESTED ← (оператор/funding-флип)
                                                     ↓
                                                 EXITING → RECONCILING → CLOSED
любой этап при невосстановимой проблеме → DEGRADED;  вход без fills → FAILED
```

| Состояние | Что происходит | Что делать оператору |
|---|---|---|
| `PREPARING` | preflight: плечо, режимы, проверки | ничего |
| `OPENING` | обе ноги отправлены **одновременно** | ничего |
| `PARTIALLY_HEDGED` | ноги наполнились неравно → repair: 1 добор отстающей, затем reduce лишнего | ничего (автоматика) |
| `HEDGED → MONITORING` | ноги выровнены, штатное сопровождение | наблюдать |
| `EXITING` | координированное закрытие: обе ноги reduce-only IOC, недобранная нога досылается | ничего |
| `RECONCILING` | сверка после закрытия | ничего |
| `CLOSED` / `FAILED` | финал | — |
| **`DEGRADED`** | **требуется человек**: дельта не выровнена или исход ноги неизвестен | см. ниже |

**Действия при DEGRADED** (придёт CRITICAL-уведомление):
1. Откройте обе биржи и сверьте фактические позиции/ордера с данными Mini App.
2. Если ноги на биржах реально сбалансированы — нажмите Close в Mini App: движок закроет
   позицию штатно (закрытие из DEGRADED поддерживается).
3. Если на бирже висит «лишняя» нога — закройте её reduce-only ордером вручную в кабинете
   биржи, затем Close в Mini App.
4. Причина всегда записана: `positions.exit_reason` / `position_transitions.reason` / audit_log.

**Ключевые инварианты** (что бот никогда не делает):
- не пере-отправляет ордер вслепую при таймауте/сетевой ошибке — только «запросил → решил»;
- не ставит независимые TP/SL на двух биржах (гарантированный источник расхождения ног);
- не маскирует дисбаланс: учёт по каждой ноге честный, до грамма;
- при неизвестном исходе ноги — **останавливается** (DEGRADED), а не гадает.

## 6. Режимы

| | Ордера | Биржи | Требования |
|---|---|---|---|
| `dry_run` (по умолчанию) | нет (наблюдение) | 7 mock-бирж с демо-данными | только БД |
| `dry_run` + `FANDOS_DEMO_ORDERS=1` | да, на mock | mock | демонстрация полного цикла, безопасно |
| `live` | **реальные** | реальные адаптеры из ваших ключей | владелец + master-key + ключи ≥2 бирж + NTP |

В live worker строит адаптеры **только** для бирж с активным trade-ключом. Нет двух бирж —
не стартует. Провалены предусловия — не стартует (в dry_run — продолжает наблюдение).

## 7. Аварийные механизмы

### 7.1 SAFE_HALT (автоматический)

Глобальный стоп новых входов. Включается сам при: недоступности БД (watchdog, 3 неудачных
ping подряд), превышении clock-offset в live, пробое дневного лимита (`riskSnap`), kill switch.
Хранится в БД (переживает рестарт) **и** в памяти (срабатывает мгновенно, даже если БД лежит).
Открытые позиции продолжают сопровождаться; запрещены только новые входы. Снятие — только
явное действие оператора после устранения причины.

### 7.2 KILL SWITCH (ручной, необратимый без повторного ввода ключей)

Красная кнопка в Mini App (или `POST /api/killswitch`). Последовательность жёсткая:
1. **немедленно SAFE_HALT** (торговля стоит, даже если следующие шаги сбоят);
2. **отзыв всех API-ключей** в боте (live-адаптеры больше не построятся);
3. запись `KILL_SWITCH` в журнал ротаций + audit.

Когда жать: подозрение на компрометацию ключей/сервера, неадекватное поведение бота, любой
«не понимаю, что происходит». Восстановление: устранить причину → заново ввести ключи
(желательно новые, старые удалить на биржах) → снять SAFE_HALT.

### 7.3 Двухфакторка (2FA)

Задайте `FANDOS_2FA_SECRET` в окружении server — тогда kill switch и сохранение ключей
требуют заголовок/поле «2FA-код» (Mini App спросит сам). Код сравнивается constant-time и
никогда не логируется. Пусто — 2FA выключена (только для разработки).

### 7.4 Withdrawal circuit breaker

N сбоев вывода подряд → автоматическая блокировка дальнейших выводов по этому маршруту до
ручного сброса. Защита от «долбёжки» в сломанный/подменённый маршрут.

## 8. Уведомления

Telegram-сообщения от бота: открытие/закрытие позиции (INFO), незавершённое закрытие (ERROR),
DEGRADED / неизвестное состояние ноги / SAFE_HALT (CRITICAL). Повторы подавляются
(rate-limit по ключу события). CRITICAL — это «посмотри сейчас».

## 9. HTTP API

Все эндпоинты (кроме `/api/auth` и служебных) требуют `Authorization: Bearer <token>`.
Мутации — заголовок `X-Idempotency-Key` (уникальная строка; повтор с тем же ключом → 409).
Критичные мутации при включённой 2FA — заголовок `X-2FA-Code`.

| Метод | Путь | Назначение |
|---|---|---|
| POST | `/api/auth` | вход: `{initData}` из Telegram → `{token, expiresAt, user}`; авто-claim владельца |
| GET | `/api/status` | состояние системы, режим, число открытых позиций |
| GET | `/api/candidates` | кандидаты сканера |
| GET | `/api/settings` | текущие HOT-настройки `{payload, version}` |
| PUT | `/api/settings` | сохранить настройки (с `version`; конфликт → 409) |
| POST | `/api/positions/{id}/close` | запрос закрытия позиции (через outbox) → 202 |
| GET | `/api/credentials` | список ключей (отпечатки, без секретов) |
| POST | `/api/credentials` | сохранить ключ: `{exchange, kind, apiKey, apiSecret, passphrase?}` → 201 `{fingerprint}` |
| DELETE | `/api/credentials/{exchange}/{kind}` | отозвать ключ → 204 |
| GET | `/api/killswitch` | статус: `{halted, reason}` |
| POST | `/api/killswitch` | активировать: `{reason}` (+2FA) → 200 |
| GET | `/` | Mini App (webapp) |
| GET | `/healthz` `/readyz` `/metrics` | служебные (на server — эти же, на worker — на `PROM_ADDR`) |

Пример ручного вызова:

```bash
TOKEN=$(curl -s -X POST localhost:8080/api/auth -d '{"initData":"..."}' | jq -r .token)
curl -s -H "Authorization: Bearer $TOKEN" localhost:8080/api/status | jq
```

## 10. Метрики

`GET /metrics` (Prometheus text): `ws_message_lag_ms`, `order_ack_latency_ms`,
`execution_leg_skew_ms` (рассинхрон ног при входе!), `delta_mismatch_duration_ms`,
`scanner_eligible_candidates`, `outbox_unprocessed`, `clock_offset_ms`,
`circuit_breaker_trips_total`, `orders_placed_total{exchange,mode}`, `reconnects_total{exchange}`.

`GET /healthz` — liveness (всегда 200 + JSON всех проверок).
`GET /readyz` — readiness: 503, если провалена **критичная** проверка (БД; в live — master-key).
На что смотреть в первую очередь: `clock_offset_ms` (live), `outbox_unprocessed` (растёт →
worker не разбирает события), `delta_mismatch_duration_ms` (ненулевой долго → беда с ногой).

## 12. Логи

Структурированный JSON (slog), секреты и 2FA-коды вырезаются redaction-фильтром.
Ключевые события: `eligible candidate`, `position opened`, `close requested`,
`position closed`, `position restored`, `precondition failed`, `SAFE_HALT`.
Корреляция — по `position_id`. Быстрые фильтры:

```bash
./bin/worker 2>&1 | grep '"msg":"position'     # жизнь позиций
./bin/worker 2>&1 | grep -E 'ERROR|CRITICAL'   # только проблемы
```

Полная история — в БД: `positions`, `position_transitions` (каждый переход с причиной),
`orders`, `fills`, `audit_log` (все действия оператора и системы).

## 13. Сценарии

**Поменять плечо / порог входа** — Mini App → Settings → изменить → Save. Применится сразу.

**Закрыть позицию сейчас** — Positions → Close. Идёт синхронное закрытие обеих ног; при
частичном исполнении бот досылает остаток; итог — CLOSED или (при проблеме) DEGRADED с
уведомлением.

**Срочно остановить всё** — KILL SWITCH (раздел 7.2). Если Mini App недоступен:
`POST /api/killswitch` curl-ом, или в крайнем случае — удалить/отключить API-ключи в
кабинетах бирж и погасить worker.

**Добавить биржу** — введите её trade-ключ в Mini App → перезапустите worker (список
live-адаптеров строится на старте).

**Ключ скомпрометирован** — Revoke в Mini App → удалить ключ на бирже → создать новый →
ввести заново. При любых сомнениях перед этим — kill switch.

**Посмотреть PnL позиции** — карточка позиции (funding-PnL, комиссии); детально — таблицы
`fills` и `funding_payments` в БД.

**Позиция в DEGRADED** — раздел 5, блок «Действия при DEGRADED».

**Снять SAFE_HALT** — устранить причину (см. уведомление/`system_locks.reason`) →
в текущей версии: снять замок в БД `UPDATE system_locks SET engaged=false WHERE
lock_name='SAFE_HALT';` и перезапустить worker (кнопка Resume в Mini App — в планах).

**Перенести на другой сервер** — перенесите PostgreSQL (pg_dump/restore) + `MASTER_KEY` +
env; код — `git clone`; состояние восстановится из БД (включая точные объёмы ног).

## 14. Ограничения

Осознанные упрощения текущей версии — проверить/учесть перед боевым использованием:

- **Withdrawal/transfer-эндпоинты** части бирж помечены `TODO:VERIFY` в коде — сверьте с
  актуальной документацией **вашей** пары бирж до включения `rebalanceEnabled`.
- **Приватные WS** (ордера/позиции в реальном времени) полностью реализованы у Binance/Bybit;
  у остальных пяти состояние ордеров подтверждается REST-запросами (медленнее на секунды,
  но безопасно за счёт QUERY_THEN_DECIDE).
- **2FA** — shared-secret заглушка; полноценный TOTP — в планах (интерфейс готов).
- **Снятие SAFE_HALT** — пока SQL-командой (см. 13).
- Ставки комиссий бирж заданы консервативной константой оценки — уточните под свои
  VIP-уровни (влияет только на отбор кандидатов, не на исполнение).
- Перед LIVE обязательно пройдите чек-лист в [`USAGE.md`](USAGE.md#11-чек-лист-безопасности)
  и месяц DRY_RUN — это дешевле любого бага.
