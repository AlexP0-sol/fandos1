# ADR-0001: Single-tenant v1, таблица `users` сохранена

- **Статус:** Accepted
- **Дата:** 2026-07-16
- **Связанные разделы промпта:** 15, 16 (v2: шапка «Single-tenant v1»)

## Контекст

В разделах 15 и 20 мастер-промпта v1/v2 явно декларируется: система предназначена для **одного пользователя**, запускающего backend на отдельном постоянно работающем компьютере. При этом раздел 16 перечисляет таблицу `users` (множественное число) в схеме БД. Это противоречие: «single-user» vs `users`.

Параллельно бизнес-потребность на будущее — возможность превратить систему в managed-продукт для нескольких независимых пользователей.

## Рассмотренные варианты

1. **Удалить `users`, сделать жёстко single-user** (одна строка в settings-таблице).
   - Минус: дорогое изменение схемы при переходе к multi-tenant; потеря history-связи.
2. **Сделать полноценный multi-tenant в v1** (registration, изоляция данных на tenant_id, биллинг).
   - Минус: значительное расширение поверхности атаки и сложности; противоречит границе первой версии (раздел 1.1) и принципу приоритета надёжности.
3. **Single-tenant runtime + таблица `users`, подготовленная под multi-tenant.** ← выбрано

## Решение

- Система в v1 работает с **единственным tenant**. Всё runtime-поведение (risk, execution, credentials) оперирует одним `tenant_id` (фиксированная константа `DEFAULT_TENANT`).
- Таблицы `users`, `telegram_sessions`, `exchange_credentials` сохраняются и имеют `tenant_id`/`user_id` первичные/ссылочные ключи — схема подготовлена под будущее multi-tenant расширение.
- Никакого UI регистрации, login, переключения аккаунтов в v1 не реализуется.
- Авторизация через Telegram initData + allowlist Admin IDs (раздел 13.4) остаётся единственным gate к этому единственному tenant.
- Все risk/execution запросы выполняются в рамках `DEFAULT_TENANT`; cross-tenant доступ невозможен, т.к. tenant один.

## Последствия

- ✅ Нет противоречия «single-user vs `users`»: `users` — это схема «на вырост», runtime — single-tenant.
- ✅ Дешёвый будущий переход к multi-tenant: добавить tenant-routing в middleware, не меняя ядро.
- ⚠️ Требуется дисциплина: не хардкодить `user_id=1` в бизнес-логике; всегда явно использовать `DEFAULT_TENANT` константу, чтобы будущий переход был механическим.
- ⚠️ В документации и UI явно указывать single-tenant, чтобы не создавать ложных ожиданий.

## Критерий приёмки

- Миграция `users` создаёт ровно одну строку-владельца (seed).
- Все доменные сущности ссылаются на tenant/owner через поле, не через захардкоженный assumption.
- В ADR-журнале зафиксировано решение.
